Project Overview

The NSCS Linux Audit & Monitoring System is a fully automated, multi-phase infrastructure audit platform built entirely in Bash and Python. It was designed to replace manual system inspection with a scheduled, self-reporting pipeline that runs without human intervention.

The system follows a modular architecture where each phase is an independent script that can run standalone or as part of the automated pipeline. A custom Python/Tkinter GUI (CustomTkinter) provides a desktop interface with live matrix rain, animated terminal output, and a scrollable module sidebar — all themed in the hacker aesthetic.

Design Principles
01
Modularity
Each script is fully independent. Any module can be run alone or chained through the cron runner. No hard dependencies between phases.
02
Dual Mode
Every script detects --gui flag. In GUI mode: auto-save, skip prompts, suppress colors. In terminal mode: full interactive experience.
03
Responsive UI
Terminal width is detected via tput cols and clamped 40–120. ASCII banners only render if TW≥72. Progress bars adapt to available width.
04
Fail-Safe
Every command has a fallback. Missing modules trigger demo mode. SSH timeouts are capped at 5 seconds. Cron failures trigger email alerts.
05
Zero Dependencies
Core modules use only standard Linux tools: bash, python3, ssh, scp, cron. No extra packages required.
06
Multi-Format Output
Reports generated in 6 formats: TXT (short+full), JSON (short+full), HTML dashboard with matrix rain, and PDF via wkhtmltopdf.
Project Phases
Phase 1
Hardware & Software Audit
Collects CPU, RAM, GPU, disk, network, BIOS data via dmidecode, lscpu, free, lspci, lsusb. Software audit reads OS, kernel, services, open ports, packages, firewall status. Output saved as structured JSON.
Phase 2
Report Generation
Parses the JSON outputs from Phase 1 using pure-bash JSON extraction (no jq required). Generates short summary and full detailed versions in TXT, JSON, HTML, and PDF. HTML reports embed live matrix rain canvas.
Phase 3
Email Transmission
Uses Python's built-in smtplib to open a direct TLS connection to Gmail (port 587, STARTTLS). Attaches all report files as MIME attachments. Credentials saved in email.conf (chmod 600). GUI mode reads config silently.
Phase 4
Cron Automation
Interactive schedule picker (8 presets + custom expression). Creates nscs_cron_runner.sh wrapper with full logging, timeout handling (300s), exit code checking, and failure alert emails. Log rotation keeps last 1000 lines.
Phase 5
Remote Monitoring
Generates RSA 4096-bit SSH key pair. Reads hosts.conf for target machines. For each host: ping check → SSH metrics collection (CPU/RAM/disk in one round-trip) → SCP push/pull reports → centralized JSON + TXT summary. Demo mode with simulated hosts when no real hosts configured.

SSH Remote Monitoring Commands
BASH — SSH KEY SETUP
# Generate RSA 4096-bit key pair (dedicated monitoring key)
ssh-keygen \
  -t rsa \                    # RSA algorithm
  -b 4096 \                   # 4096-bit strength
  -f ~/.ssh/nscs_monitor_rsa \ # dedicated key file
  -C "nscs_monitor@kali" \   # label embedded in key
  -N ""                       # no passphrase (automation)

# Copy public key to remote machine (one-time setup)
ssh-copy-id -i ~/.ssh/nscs_monitor_rsa.pub kali@192.168.1.10

# Collect ALL metrics in ONE SSH connection (efficient)
REMOTE_DATA=$(ssh \
  -i ~/.ssh/nscs_monitor_rsa \
  -o ConnectTimeout=5 \      # fail fast — don't hang
  -o BatchMode=yes \         # never ask for password
  -o StrictHostKeyChecking=accept-new \ # MITM protection
  -o LogLevel=ERROR \        # suppress SSH messages
  kali@192.168.1.10 bash <<'CMD'
    CPU=$(top -bn1 | grep "Cpu(s)" | awk '{print 100-$8}' | cut -d. -f1)
    RAM=$(free | awk '/^Mem:/{printf "%.0f", $3/$2*100}')
    DISK=$(df / | awk 'NR==2{print $5}' | tr -d '%')
    echo "CPU=$CPU|RAM=$RAM|DISK=$DISK"
CMD)

# SCP — push report to remote
scp -i ~/.ssh/nscs_monitor_rsa report_full.txt kali@192.168.1.10:/tmp/

# SCP — pull remote audit report
scp -i ~/.ssh/nscs_monitor_rsa kali@192.168.1.10:~/nscs_os_project/reports/*.json ./remote_reports/

The expression 0 4 * * * means: "At minute 0 of hour 4, every day of every month, every weekday" — i.e., daily at 04:00 AM.

Cron Runner — Failure Handling Logic
BASH — nscs_cron_runner.sh
# run_module: executes one script and handles its exit code
run_module() {
  local script="$1"  label="$2"

  # timeout 300 = kill process if it runs longer than 5 minutes
  # >> $LOG_FILE 2>&1 = capture both stdout AND stderr to log
  timeout 300 bash "$MODULES_DIR/$script" --gui >> "$LOG_FILE" 2>&1
  local exit_code=$?

  # $? contains exit code: 0=success, 124=timeout, other=error
  if [ $exit_code -eq 0 ]; then
    log_ok "$label completed successfully"
    SUCCESS_MODULES+=("$label")

  elif [ $exit_code -eq 124 ]; then
    log_error "$label TIMED OUT after 300s"
    FAILED_MODULES+=("$label:TIMEOUT")

  else
    log_error "$label FAILED — exit code $exit_code"
    FAILED_MODULES+=("$label:exit $exit_code")
  fi
}

# After all modules run, send alert if any failed
if (( ${#FAILED_MODULES[@]} > 0 )); then
  send_failure_alert_email   # notifies admin immediately
fi

# Exit code = number of failures (0 = total success)
exit ${#FAILED_MODULES[@]}

Area Threat Mitigation Applied Level
SSH Authentication Password brute-force, credential theft RSA 4096-bit key pair. No password ever transmitted. BatchMode=yes prevents prompt fallback. HIGH
SMTP Credentials Credential exposure in plain text email.conf saved with chmod 600 (owner read-only). Gmail App Password used — not real password. 2FA required on Google account. HIGH
MITM Attacks Attacker impersonates remote host StrictHostKeyChecking=accept-new: saves host fingerprint on first connect, rejects if it changes on future connections — detects replacement or interception. HIGH
Transport Encryption Data interception in transit SSH uses AES-256-GCM encryption. SMTP uses STARTTLS (TLS 1.2+). SCP is SSH-based. All data in transit is encrypted. HIGH

SMTP Security — Gmail App Password Flow
PYTHON — smtplib TLS handshake
# 1. Open plain TCP connection to Gmail port 587
server = smtplib.SMTP("smtp.gmail.com", 587, timeout=15)

# 2. Announce ourselves (required by SMTP protocol)
server.ehlo()

# 3. UPGRADE the connection to encrypted TLS
#    From this point, ALL data is AES-256 encrypted
server.starttls(context=ssl.create_default_context())
server.ehlo()   # re-announce after TLS upgrade

# 4. Authenticate with App Password (NOT your real Gmail password)
#    App Password = 16-char code generated by Google
#    Can be revoked without changing your real password
server.login("your@gmail.com", "abcd efgh ijkl mnop")

# 5. Send the email with attachments
server.sendmail(smtp_from, smtp_to, msg.as_string())

Challenges Encountered
Challenge 1 — GUI Subprocess Encoding Crash
PROBLEM
The Python GUI crashed with UnicodeDecodeError: 'utf-8' codec can't decode byte 0xe2 when launching bash modules. The subprocess used text=True which applied strict UTF-8 decoding — but bash scripts output raw Unicode box-drawing characters (█, ▶, ═) that failed Python's strict decoder.
SOLUTION
Switched to reading raw bytes (proc.stdout.read(1)) and decoding with errors="replace". Added TERM=dumb and NO_COLOR=1 environment variables to minimize ANSI output. Added a regex to strip remaining escape codes: \x1b\[[0-9;]*[mABCDEFGHJKST...]. Added a garbled-character filter that drops any line where more than 40% of characters are Unicode replacement chars (U+FFFD).
Challenge 2 — Tkinter Internal Method Name Collision
PROBLEM
The GUI crashed immediately on startup with TypeError: MatrixRain._setup() takes 1 positional argument but 3 were given. The custom MatrixRain class inherited from tk.Canvas and defined a method called _setup() — which collided with a tkinter internal method of the same name that gets called during widget initialization with arguments.
Renamed _setup() to _init_columns() throughout the class to avoid the name collision with tkinter's internal namespace.
Challenge 3 — GUI Script Output Corruption (Garbled Blocks)
PROBLEM
The terminal output panel in the GUI showed broken green blocks and overlapping characters when running modules. The root cause was the typewriter effect: stream_line() wrote text character-by-character, calling update_idletasks() after every single character. This forced tkinter to redraw mid-line while the script was simultaneously pushing new bytes — creating partial render collisions.
Replaced stream_line() with atomic append() (writes full line instantly). Added clean per-line sleep delays instead: head lines 120ms, OK lines 60ms, data lines 30–55ms. Result: same hacker drip effect, zero corruption.
Challenge 4 — Scripts Breaking on Narrow GUI Panel
PROBLEM
Bash scripts read terminal width with tput cols and used it to print full-width borders and ASCII banners. Inside the GUI subprocess, tput cols returned the GUI panel width (~50 chars) instead of a full terminal width. The 68-character ASCII art banners wrapped mid-character, producing those green block artifacts seen in screenshots.
Added responsive design to all 5 scripts: TW clamped between 40–120. print_banner() checks TW >= 72 before printing ASCII art — falls back to compact [ NSCS — MODULE ] on narrow screens. All repeat_char() calls guarded with (( count <= 0 )) && return. Progress bars calculated as TW - label_len - 14 with floor of 5.
Challenge 5 — Interactive Prompts Blocking GUI
PROBLEM
Scripts like send_reports.sh used read -r to ask for email credentials and confirmation. When launched from the GUI subprocess (which has no stdin), these read commands blocked indefinitely — the script appeared to hang with no output and never finished.
Added --gui flag detection in all scripts. In GUI mode: skip all read prompts, auto-save JSON, read credentials from email.conf. For send_reports.sh: shows a clear error if no config exists with instructions to run once in terminal. For setup_cron.sh: uses safe default schedule in GUI mode.
Challenge 6 — npm Permission Error (Claude Code Install)
PROBLEM
npm install -g @anthropic-ai/claude-code failed with EACCES: permission denied, mkdir '/usr/local/lib/node_modules'. npm's global install directory required root access which is bad practice to grant.
Configured npm to use a user-local directory: mkdir ~/.npm-global && npm config set prefix '~/.npm-global' then added ~/.npm-global/bin to $PATH in .bashrc. This allows global npm installs without sudo.
